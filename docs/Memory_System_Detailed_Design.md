# Project Crystal: 个性化记忆系统 (PMS) 详细设计说明书

## 1. 引言

### 1.1 项目背景
在当今的知识管理与 AI 协作领域，通用的 AI 助手往往面临“冷启动”和“语境缺失”的问题。用户在每次开启新的写作任务时，都需要反复向 AI 解释背景资料、项目细节以及特定的语言风格。为了解决这一痛点，Project Crystal 引入了**个性化记忆系统 (Personalized Memory System, PMS)**。

### 1.2 系统目标
PMS 的核心目标是构建一个“可生长的、数字化的第二大脑”。它能够自动从用户的笔耕不辍中汲取养分，将碎片化的信息提炼为结构化的知识，并在恰当的时机（如续写、总结或对话时）主动召回，从而实现真正的“AI 辅助思考”。

## 2. 设计哲学与参考依据

### 2.1 灵感来源：Memos (原子化与隐私)
*参考对象：Use-Memos (GitHub 开源项目)*
- **核心理念**：Memos 强调“轻量级卡片式捕捉”。
- **系统借鉴**：PMS 采用了 Memos 的原子化存储方案。我们不存储冗长的文档作为记忆，而是将记忆拆解为一个个独立的“知识原子”（Memorandum Atoms）。这样做的好处是检索粒度极细，能够精准匹配当前的写作段落，避免不相关的长文本干扰 AI 判断。

### 2.2 灵感来源：Mem.ai (主动式语境注入)
*参考对象：Mem.ai (商业化 AI 笔记工具)*
- **核心理念**：Mem 最大的特色是其自组织能力和“预测性”内容推荐。
- **系统借鉴**：PMS 实现了**主动召回 (Proactive Recall)** 机制。用户不需要手动检索记忆，系统会根据当前的光标上下文，实时从 MySQL 库中“捞出”相关的项目事实或风格偏好，并通过系统级提示词（System Prompt）注入到大模型中，实现“未卜先知”的协作感。

### 2.3 灵感来源：Obsidian (关系与溯源)
*参考对象：Obsidian (双链笔记)*
- **核心理念**：万物皆有联系，知识不应是孤岛。
- **系统借鉴**：PMS 为每一条记忆保留了 `SourceID`。所有的 AI 推荐都可以追溯到原始的 Crystal 文档。这解决了 AI 幻觉问题，让用户能够随时通过点击链接查看记忆的原始出处，确报真实可靠。

## 3. 系统总体架构

系统采用清晰的四层架构设计，确保了组件的高内聚与低耦合：

### 3.1 采集层 (Ingestion Layer)
- **职责**：监听 Crystal 文档的变更。
- **技术栈**：Go Goroutines, Gin Middleware。
- **机制**：当 `UpdateCrystal` 接口被成功调用后，系统会启动异步协程，避免阻塞用户保存文档的主流程。

### 3.2 提取引擎 (Extraction Engine)
- **职责**：利用 AI 从原始文本中提炼“核心知识”。
- **维度一：事实维度 (Fact Dimension)**：识别项目名、人名、时间节点、核心业务逻辑。
- **维度二：风格维度 (Style Dimension)**：识别用户的写作口吻（如专业、口语化）、句式习惯、排版偏好（如喜欢用列表还是段落）。

### 3.3 存储层 (Persistence Layer)
- **职责**：实现知识的持久化与快速检索。
- **当前方案**：MySQL 8.0。利用关系型数据库的高并发写入能力。
- **未来方案**：Graph Database (Neo4j)。通过关系的拓扑结构实现知识的联想。

### 3.4 召回层 (Retrieval Layer)
- **职责**：在 AI 调用时实现语境匹配。
- **算法**：关键词加权检索 + 向量语义检索 (Future)。
- **输出**：生成一段结构化的“上下文补充包”，注入给后端 AI Handler。

## 4. 详细数据模型设计 (MySQL)

为了支持未来的图数据库迁移，我们在 MySQL 的表结构设计上做了高度的抽象化。

### 4.1 `memories` 表结构定义

| 字段名 | 数据类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | Primary Key, Auto Increment | 唯一标识符 |
| `category` | `VARCHAR(20)` | NOT NULL | 类别：`fact` (事实), `style` (风格) |
| `content` | `TEXT` | NOT NULL | 记忆的原始文本内容 |
| `entity` | `VARCHAR(100)` | INDEX | 该记忆归属的实体（如 "Project_Alpha"） |
| `source_id` | `BIGINT` | FOREIGN KEY (crystals.id) | 来源文档 ID |
| `importance` | `TINYINT` | DEFAULT 3 | 重要程度评分 (1-5) |
| `version` | `INT` | DEFAULT 1 | 知识版本号，处理更新冲突 |
| `created_at` | `DATETIME` | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | `DATETIME` | ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

### 4.2 索引策略
- **Entity 索引**：用于快速通过关键词定位相关事实。
- **Temporal 索引**：最新的文档产生的内容往往具有更高的时效性，优先展示。
- **Category 索引**：区分风格注入与事实注入。

## 5. 提取引擎 (Extraction Engine) 详细逻辑

这是系统的“大脑”。它负责在后台默默地将非结构化文本转化为结构化知识。

### 5.1 事实维度提取 (Facts Extraction)
当系统判定文档内容发生显著变化时，会构造如下 Prompt 请求 SiliconFlow：
> "作为一名专业的研究助理，请分析提供的文本，提取出其中包含的项目名称、关键人物、技术栈、最后期限、核心目标。请直接返回 JSON 格式，不包含任何解释文字。"

**JSON 示例输出：**
```json
[
  {"entity": "Lumina", "fact": "Lumina 项目将于五月底交付，重点是重构登录链路。"},
  {"entity": "张经理", "fact": "张经理负责 Lumina 项目的预算审批。"}
]
```

### 5.2 风格维度提炼 (Style Profiling)
除了硬性知识，AI 还必须学习如何模仿你的笔迹。
> "请分析此段文本的写作风格。请从：语气（学术/随意）、表达偏好（长难句/短句）、常用排版方式三个角度各总结 20 字以内的一句话。"

**风格档案示例：**
- **语气**：严谨且不失温度，经常使用肯定句式。
- **偏好**：极度依赖无序列表（Bulleted list）展示数据。
- **高频词**：习惯使用“底层逻辑”、“闭环”等互联网术语。

## 6. 召回与注入 (Recall & Injection) 详细逻辑

这是 PMS 系统发挥价值的“最后一公里”。

### 6.1 召回触发
当用户执行以下操作时触发召回：
1. **自动补全 (Tab)**：实时召回风格指纹。
2. **AI Action (如总结)**：召回上下文事实。

### 6.2 召回优先级算法
1. **精准匹配**：如果文档标题包含 "Project Astra"，则优先加载 `entity="Astra"` 的记忆。
2. **时间优先**：加载最近 24 小时内更新的所有活跃记忆。
3. **重要性加权**：权重高于等于 4 的关键记忆作为“全局上下文”常驻。

### 6.3 注入示意 (Prompt Engineering)
后端在调用大模型前，会将召回的记忆动态拼接到 `System Role` 中：
```text
Role: 你是 Crystal 系统的高级写作助理。
### 当前用户记忆补充 ###
[事实]: 用户正在进行 Lumina 项目，目标是登录重构。
[风格]: 你的回答应多使用列举法，语气要专业简练。
#######################
当前输入内容: ...
```

## 7. 前端交互与状态设计

虽然 PMS 是全自动的，但必须给用户以明确的反馈。

### 7.1 提取状态反馈 (The Pulsing Icon)
在文档保存后，编辑器底部会显示一个细微的“闪烁”状态（Aura 动画），提示用户：“记忆提取引擎正在提取新知识”。这种设计参考了 Mem.ai 的无感同步感。

### 7.2 记忆面板 (展望)
未来将在左侧或右侧侧边栏提供一个“记忆之井 (Memory Well)”界面，显示当前 AI 已经掌握的相关知识点，并允许用户手动：
- **置顶**：让某条记忆变为全局常驻。
- **修正**：如果 AI 提取错误，用户可以手动编辑。
- **删除**：彻底抹除某条特定信息的记忆。

## 8. 安全性与隐私策略

- **本地化先行**：所有核心记忆库存储在用户本地的 MySQL 实例中，不存储在云端。
- **端到端加密 (计划中)**：在将内容发送给 SiliconFlow 提取前，通过预处理器剔除敏感词（如手机号、身份证号）。
- **用户授权隔离**：如果是多用户环境，每个用户的记忆 ID 是严格隔离的，确保隐私不泄露。

## 9. 扩展性：向图数据库 (Graph DB) 进发

当前的 MySQL 结构虽然简单，但在处理**深度关系**时存在局限。

### 9.1 图谱节点定义
- **Node(Topic)**: Lumina Project
- **Node(Person)**: 张经理
- **Node(Tech)**: React 19

### 9.2 图谱边 (Relationship) 定义
- `(张经理)-[LEADS]->(Lumina Project)`
- `(Lumina Project)-[USE_TECH]->(React 19)`

迁移到图数据库后，当用户写到“React 19”时，AI 能够通过路径联想到“Lumina 项目”及其负责人“张经理”，实现跨文档、跨维度的深度智能，这正是类似 Heptabase 和 Roam Research 追求的究极愿景。

## 10. 接口定义 (API Specification)

### 10.1 记忆提取触发接口 (Internal)
- **Endpoint**: `POST /api/v1/internal/memory/extract`
- **Request**: `{ "crystal_id": 101, "content": "..." }`
- **Response**: `202 Accepted`

### 10.2 记忆召回接口
- **Endpoint**: `GET /api/v1/memory/recall`
- **Query**: `context=Project_Astra&limit=5`
- **Response**: `[{"category": "fact", "content": "..."}, ...]`
